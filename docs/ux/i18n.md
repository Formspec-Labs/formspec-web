# I18n

The demo form uses Formspec Locale sidecars loaded into the active engine.

## Locale Sidecars

The respondent runtime loads Locale Documents through the active `DefinitionSource`.
The demo composition seeds English and Spanish Locale Documents from
`src/demo/locales.ts`. The HTTP composition extracts concrete server-supplied
Locale Documents from the runtime payload fields `locales`, `locale_documents`,
or `localeDocuments`.

Locale Documents include:

- `$form.title`.
- `$form.description`.
- Field labels and hints.
- Option-set labels.

## Locale Toggle

The respondent shell reads the locale list from `x-formspec-locales` metadata and calls `engine.setLocale(code)` when the respondent switches language.

The React renderer updates labels through engine locale signals. The shell also re-resolves the form title and description after switching.

## FEL Interpolation

Locale strings can include Formspec Expression Language interpolation. The demo includes an interpolated preferred-language hint that reads from `$applicant.fullName`, proving localized strings can react to response data.

## Production Runtime Payload

FW-0019 is adapter-proven at the web boundary: `HttpDefinitionSource` reads the
published runtime payload once, returns the Definition, and exposes any concrete
Locale Documents carried beside it. Legacy `locale_refs` string arrays are
treated as references only; they do not synthesize translations. If a production
server still emits only `locale_refs`, the runtime falls back to inline Definition
strings for those locales until the server includes concrete Locale Documents in
the payload.
